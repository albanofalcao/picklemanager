🔗 Relacionamentos entre Entidades

Entidade Cardinalidade Entidade Descrição
perfis 1 → N usuarios Um perfil pode ter vários usuários
alunos 1 → N matriculas Um aluno pode ter várias matrículas ao longo do tempo
planos 1 → N matriculas Um plano pode estar vinculado a várias matrículas
professores 1 → N aulas Um professor pode ministrar várias aulas
arenas 1 → N aulas Uma arena pode receber várias aulas
aulas 1 → N presencas Uma aula tem um registro de presença por aluno
alunos 1 → N presencas Um aluno tem registros de presença em múltiplas aulas
arenas 1 → N eventos Uma arena pode sediar vários eventos
arenas 1 → N manutencao Uma arena pode ter vários chamados de manutenção
arenas 1 → N manutencao_prev Uma arena pode ter várias manutenções preventivas agendadas
aulas N ↔ M alunos Muitos alunos em muitas aulas — via tabela presencas
alunos N ↔ M planos Muitos alunos em muitos planos — via tabela matriculas
Banco de dados: localStorage do navegador · Prefixo de todas as chaves: pm_
Formato: JSON serializado por chave · Cada registro possui id (base36 + random), createdAt e updatedAt (ISO 8601) gerados automaticamente pelo Storage.
Chaves de sessão: pm_session (objeto de sessão do usuário logado) · pm_theme (light | dark) · listas_config (listas customizadas).
Integridade referencial: implementada por convenção (sem constraints nativas). FKs são armazenadas como Id + Nome (cache) para evitar JOINs no frontend.
